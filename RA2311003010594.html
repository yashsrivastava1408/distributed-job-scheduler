1. Project Objective

Implementation of a production-inspired, highly reliable distributed job scheduling platform capable of executing asynchronous background tasks across multiple workers concurrently. The system prioritizes database consistency, concurrency controls (prevention of double claims), client isolation boundaries, and real-time operational status updates.

2. System Architecture

The system is divided into three isolated layers:

  • REST API Layer: Serves user authentication, project creation, queue configurations, and job pushes.
  • Database Storage Layer: A normalized PostgreSQL database (Supabase) that holds queue configurations, jobs, batches, executions, and metrics.
  • Distributed Workers Layer: Multiple background daemon instances polling database slots concurrently, claiming jobs atomically via database transactions, and running them based on type.

4. Core Technical Specifications

4.1 Atomic Job Claiming (PostgreSQL SKIP LOCKED)

To resolve database race conditions under high workloads, claiming is processed inside an atomic transaction:

-- Atomic claim query executed within a transaction SELECT * FROM "jobs" WHERE "queue_id" = $1 AND "status" = 'queued' AND "run_at" <= NOW() ORDER BY "priority" DESC, "run_at" ASC LIMIT $2 FOR UPDATE SKIP LOCKED;

This query locks the matching job records so no other concurrent worker can see or claim them, while SKIP LOCKED ensures other workers skip past them immediately without blocking.

4.2 Multi-Tenancy & Project Boundaries

  • Dual Authentication: Web clients use JWT credentials sessions to manage projects, while client daemons submit tasks using unique API Keys (prefixed with jsk_).
  • Fast SHA-256 Lookup: API keys are stored in the database as SHA-256 hashes instead of slow bcrypt hashes, allowing direct indexed lookups.
  • Isolation Middleware: The API checks that any referenced queue or job ID matches the caller's project scope.

4.3 Stale Worker Watchdog (Reaper)

Workers update their lastHeartbeatAt timestamp every 5 seconds. If a worker crashes or drops offline, the Reaper process (running every 15 seconds) identifies workers that have missed heartbeats for over 30 seconds, updates their status to offline, and safely marks their active jobs back to a queued state so they can be retried by other workers.

4.4 Resilient Retry Controls & DLQ

  • Custom Retry Policies: Queues configure delay strategies (fixed, linear, or exponential with ±25% random jitter) to prevent thundering herd problems.
  • Dead Letter Queue: If a job's attempts exceed the maximum limits, it is marked failed and moved to the DeadLetterEntry registry, where logs are saved and users can trigger manual retries.
